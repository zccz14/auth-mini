# `examples/demo` Credentials 管理页设计

## 背景

- 当前 `examples/demo` 已提供 Setup、Email、Passkey、Session 等顶层路由，用于演示浏览器 SDK 的核心认证流程。
- 现有 demo 能展示当前 session / user 快照，但缺少一个聚焦“当前登录用户已绑定凭据”的集中管理页，无法在同一处查看邮箱、Passkey 与 Ed25519 凭据，并执行受控解绑。
- 本轮已确认设计范围仅覆盖当前已登录用户自己的凭据管理，不扩展到凭据创建、更新、登录或管理员视角。

## 目标

- 在 `examples/demo` 新增一个顶层 `Credentials` 路由，导航层级与 Setup、Email、Passkey、Session 并列。
- 提供一个沿用当前紧凑 demo / app-shell 视觉语言的凭据管理页，页面按从上到下的顺序展示 Email、Passkey、Ed25519 三个分区。
- 将主要交互从 JSON 面板切换为表格化展示，突出“查看当前绑定项 + 执行删除”这一管理动作。
- 基于当前认证态下的 `/me` 快照展示数据，并在删除成功后从服务端重新刷新 user / me 状态。
- 为该页补充覆盖路由导航、未登录态、已登录列表渲染以及删除成功流程的测试。

## 非目标

- 不在本页提供邮箱、Passkey 或 Ed25519 凭据的新建、注册、更新、重命名、登录或重新验证流程。
- 不在本页引入跨用户管理、管理员视角或任意用户凭据查询能力。
- 不把删除成功后的 UI 状态建立在乐观更新上，也不在本轮扩展批量删除、多选操作或高级筛选。
- 不改变现有 Setup、Email、Passkey、Session 页面职责，只新增一个独立的凭据管理入口。

## 决策

采用单页三分区的 Credentials 管理方案：新增 `Credentials` 顶层路由，页面垂直排列 Email、Passkey、Ed25519 三个 section；所有 section 都只读取当前 authenticated `/me` 快照中的当前用户数据；Email 区只读展示主绑定邮箱，Passkey 与 Ed25519 区通过表格列出可解绑项，并在浏览器原生确认后执行删除；删除成功后统一重新拉取 `/me`，删除失败只在对应 section 内反馈，不阻塞其他 section。

## 路由与导航

- 新增顶层路由 `/credentials`，并在应用壳顶部导航中新增 `Credentials` 入口。
- `Credentials` 在导航中的层级与 Setup、Email、Passkey、Session 完全一致，不作为子页或隐藏入口。
- 路由进入后直接渲染凭据管理页，不需要额外的 tab、嵌套路由或二级导航。

## 页面结构

### 整体布局

- 页面复用当前 demo 的紧凑 app-shell、卡片和分区语言，保持与现有页面一致的产品化观感。
- 页面主体按从上到下顺序固定为：Email、Passkey、Ed25519。
- 主要信息展示使用表格，而不是 JSON panel；只有本页必要的状态说明、错误提示和空状态文案会以文本或 alert 形式呈现。

### Email 分区

- Email 分区只表示当前用户的主绑定邮箱，数据来源为 `session.me.email`。
- 若用户已登录且存在邮箱，则以最小表格或单行表格展示，列为 `Email / Type / Status`。
- `Type` 用于表达该邮箱是当前主绑定邮箱；`Status` 用于表达只读/已绑定状态。
- Email 的 action 区域只能展示只读状态说明，例如“由 email OTP 登录管理”，不得出现删除、解绑或编辑按钮。
- 若已登录但 `session.me.email` 为空，则该分区展示“当前账户未绑定邮箱”的只读空状态，不提供任何操作。

### Passkey 分区

- Passkey 分区展示当前用户的 `webauthn_credentials`。
- 表格列固定为 `Credential ID / Created At / Action`。
- 每一行代表一条当前用户自己的 Passkey 记录。
- `Action` 只提供 unbind/delete 能力，触发前必须使用浏览器原生确认；用户取消确认时不得发起删除请求。
- 删除成功后必须重新刷新 `/me`，以服务端返回结果作为最新列表来源。

### Ed25519 分区

- Ed25519 分区展示当前用户的 `ed25519_credentials`。
- 表格列固定为 `Name / Public Key / Last Used / Created At / Action`。
- `Public Key` 在 UI 中必须截断展示，避免长文本破坏表格布局；是否显示完整值不属于本轮范围。
- `Action` 只提供 unbind/delete 能力，触发前必须使用浏览器原生确认；用户取消确认时不得发起删除请求。
- 删除成功后必须重新刷新 `/me`，以服务端返回结果作为最新列表来源。

## 数据来源与刷新模型

- 本页所有数据都来自当前 authenticated 状态下的 `/me` 快照，不额外引入新的聚合接口。
- Email 数据直接读取 `session.me.email`。
- Passkey 与 Ed25519 列表分别读取当前用户快照中的 `webauthn_credentials` 与 `ed25519_credentials`。
- 本页初次渲染时使用共享 provider 中已有的当前 user / me 快照；如 provider 支持刷新，则本页的删除成功回调必须走统一刷新路径，重新从服务端拉取 `/me`。
- 删除成功后不得只在本地数组中手动移除单项再假定状态正确，避免与服务端最终状态不一致。

## 认证态与空状态

- 本页范围仅覆盖当前登录用户自己的凭据；未认证时不得展示任何他人数据或伪造占位数据。
- 未认证时，Email、Passkey、Ed25519 三个分区都要展示 sign-in-required / empty-state 文案，明确说明需要先登录后才能查看当前账户凭据。
- 未认证时三个分区都不得出现删除、解绑等破坏性操作按钮。
- 已认证但某个分区没有数据时，该分区单独展示空状态；其他分区仍按各自真实状态渲染。

## 删除交互与错误处理

- Passkey 与 Ed25519 的删除交互必须是逐行触发，不提供批量删除。
- 删除前必须调用浏览器原生确认能力，确认文案应明确表达这是解绑/删除当前凭据的破坏性动作。
- 删除进行中时，仅锁定当前分区内相关动作，避免重复提交；不要求阻塞其他分区的查看与交互。
- 删除失败时，错误信息必须显示在对应 section 内，并且不能阻塞另外两个 section 的展示或操作。
- Passkey 删除失败只影响 Passkey 分区；Ed25519 删除失败只影响 Ed25519 分区；Email 分区始终保持只读。

## 文案与表现边界

- 页面文案保持当前 demo 的简洁、产品化风格，避免引入大段解释性文档文本。
- 表格时间字段沿用 demo 当前一致的可读化时间展示方式；若现有页面暂无统一格式，实现时应选择一个稳定、可测试的展示格式，但不在本 spec 中扩展新的时间格式化能力。
- `Credential ID` 与 `Public Key` 都允许为适配表格布局做截断展示；其中 `Public Key` 截断是本轮明确要求，`Credential ID` 是否截断可按实际布局需要决定，但不改变列名与数据语义。

## 测试与验证范围

- 路由 / 导航测试必须覆盖新增 `Credentials` nav entry，并验证 `/credentials` 路由可以正确渲染页面。
- 页面测试必须覆盖未认证时三个分区都展示空状态 / 需登录提示，且没有删除按钮。
- 页面测试必须覆盖已认证时 Email、Passkey、Ed25519 的列表 / 行内容渲染。
- 页面测试必须覆盖 Passkey 删除成功：原生确认通过、删除请求发出、随后刷新 `/me`、页面反映刷新后的结果。
- 页面测试必须覆盖 Ed25519 删除成功：原生确认通过、删除请求发出、随后刷新 `/me`、页面反映刷新后的结果。
- 页面测试必须覆盖 Email 分区为只读展示，且不存在删除按钮。

## 风险与控制

- 风险：实现时直接复用 JSON panel，导致页面管理语义不清晰、与已批准设计不一致。
  - 控制：spec 明确要求主交互以表格为核心，JSON panel 不作为本页主要展示方式。
- 风险：删除成功后只做本地乐观移除，导致列表与服务端 `/me` 快照不一致。
  - 控制：删除成功后强制重新刷新 `/me`，以服务端返回状态为准。
- 风险：某一分区删除失败后把整页置为错误态，破坏其他分区的可用性。
  - 控制：错误信息限定在对应 section 内，其他 section 独立渲染与操作。
- 风险：未认证时仍暴露删除按钮或伪造数据，削弱“仅当前登录用户本人数据”的边界。
  - 控制：未认证时三个分区统一进入 sign-in-required / empty-state，且无任何破坏性操作。
