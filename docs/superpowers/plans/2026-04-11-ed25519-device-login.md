# ED25519 Device Login Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add ED25519 device credential CRUD plus challenge-based device login that mints the existing AuthMini session token pair with `amr = ['ed25519']`.

**Architecture:** Keep ED25519 parallel to WebAuthn instead of abstracting both into one generic credential layer. Add new SQLite tables and a focused `src/modules/ed25519/` module for credential storage and challenge verification, then thread the resulting session auth method through the existing session/JWT pipeline and `/me` response.

**Tech Stack:** TypeScript, SQLite, Hono, Node `crypto` Ed25519 verify helpers, Vitest integration/unit tests, existing JWKS/session infrastructure

---

- [ ] **Step 5: Implement signature verification and session minting**

Extend `src/modules/ed25519/service.ts` with:

```ts
export async function verifyAuthentication(
  db,
  input: {
    requestId: string;
    signature: string;
    issuer: string;
    logger?: AppLogger;
  },
) {
  return {
    session_id: 'session-id',
    access_token: 'access-token',
    token_type: 'Bearer' as const,
    expires_in: 900,
    refresh_token: 'refresh-token',
  };
}
```

Implementation rules:

- verify the signature over the raw `challenge` string only
- use Node `crypto.verify(null, Buffer.from(challenge, 'utf8'), publicKey, signature)` with an imported Ed25519 public key derived from the stored 32-byte raw key material
- consume the challenge before returning success
- update `last_used_at` in the same success path
- mint the session with `authMethod: 'ed25519'`
- collapse all missing/expired/consumed/signature-invalid cases into `InvalidEd25519AuthenticationError`
